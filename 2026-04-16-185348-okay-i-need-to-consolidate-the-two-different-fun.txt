  Time to draft the design. I'll present it in sections and check with you after
   each. Here are the first three — the spine of the system.

  ---
  Section 1 — Overview

  The Claude Usage Optimizer is a single Next.js app that does five things in
  one process:

  1. Polls claude.ai for usage snapshots (existing collector, unchanged).
  2. Stores raw API JSON in SQLite (simplified schema per hosting doc §5.2).
  3. Computes an optimal daily send schedule from all historical usage data.
  4. Fires sends by spawning claude -p via child_process.
  5. Surfaces schedule, peak detection, and overrides on the dashboard.

  One systemd unit (claude-tracker.service), one config file, one log stream.
  The Claude Message Sender/ directory is deleted.

  ---
  Section 2 — Algorithm

  Peak detection (runs nightly at 03:00 UTC)

  1. Query all status = 'ok' snapshots, chronological.
  2. Parse raw_json at read time. For each consecutive pair, compute 5-hour
  utilization delta (reuse existing computeUsageDelta logic, now post-parse).
  3. Aggregate into 24 hourly buckets: hourlyTotal[0..23].
  4. Slide a 4-hour window across the buckets, wrapping past midnight. Pick the
  contiguous block with the max sum → peak block.
  5. Midpoint = block_start + 2 hours.

  Schedule generation

  1. Anchor fire time = midpoint + :05 safety offset (keeps the existing 5-min
  buffer).
  2. Generate the daily chain: anchor, anchor+5h, anchor+10h, anchor+15h,
  anchor+20h — all five wrap past 24h. This fixes the existing overflow bug.
  3. Apply the existing 0–5 min randomization to non-anchor fires. Anchor stays
  exact.

  Edge cases & fallbacks

  - <3 days of data → fall back to user-configured default seed (default 05:05).
  - Ties between 4-hour blocks → pick the one whose midpoint is closest to
  midday. Deterministic.
  - Midnight wrap → sliding window uses (start + i) % 24.
  - User override present → skip peak detection entirely, use override verbatim.

  ---
  Section 3 — Architecture

  claude-tracker (Next.js, single systemd unit)
  ├── Collector           (existing)  adaptive polling → writes raw_json
  ├── DB                  (refactored) SQLite, simplified schema
  ├── Scheduler           (NEW)      in-process tick loop: fires sends,
  │                                  nightly recompute
  ├── Peak detector       (NEW)      pure fn: snapshots → peak block + midpoint
  ├── Schedule generator  (NEW)      pure fn: midpoint + overrides → fire times
  ├── Sender              (NEW)      child_process spawn `claude -p`, log to DB
  ├── Query layer         (NEW)      json_extract-based read helpers
  └── Dashboard           (extended) heatmap, peak card, schedule, overrides

  New files:
  - src/lib/peak-detector.ts
  - src/lib/schedule.ts
  - src/lib/sender.ts
  - src/lib/scheduler.ts
  - src/lib/queries.ts (absorbs/replaces much of normalize.ts usage on the read
  path)

  Data flow:
  1. Collector writes snapshot → DB (unchanged).
  2. 03:00 UTC nightly: scheduler calls peak-detector → writes
  app_meta.schedule_fires, app_meta.peak_block.
  3. Each minute: scheduler checks if any fire time matches; if yes, invokes
  sender → sender spawns claude -p → writes result to DB (send_log or as a row
  in usage_snapshots with a new kind flag — TBD in Section 4).
  4. Dashboard reads DB + app_meta, renders everything.

  Startup catch-up: If the process restarts and misses a fire time by <15 min,
  fire immediately. If >15 min late, skip (the window already advanced).
