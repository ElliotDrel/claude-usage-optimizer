## Critical Integration Gaps

### Gap 1: Setup Gate Middleware Not Mounted

**Severity:** CRITICAL  
**Affected Requirements:** INSTALL-01, INSTALL-02, INSTALL-03, QUAL-02  
**Impact:** First-run setup wizard never activates. Users bypass `/setup` gate and land directly on dashboard on first boot.

**Root Cause:**  
`src/proxy.ts` contains the setup gate logic (reads `getAppMeta`, redirects to `/setup` if setup_complete ≠ 'true'), but **is never registered as Next.js middleware**.

Next.js requires middleware to be either:
- Exported from `src/middleware.ts` (app router), or
- Exported from `middleware.ts` at project root

**Current State:**
- ✓ `src/proxy.ts` exists with correct gate logic
- ✓ Phase 7 (Installer) verifies middleware.ts should exist (VERIFICATION.md line 85)
- ✗ No `src/middleware.ts` or `middleware.ts` file found

**Fix Required:**
Create `src/middleware.ts`:
```typescript
export { setupGate as middleware } from './proxy'
export const config = {
  matcher: ['/((?!api|_next/static|favicon).*)']
}
```

Or equivalently at project root as `middleware.ts`.

**Verification:** After fix, middleware runs on first request → redirects to `/setup` → user completes form → `setup_complete='true'` → redirects to `/` dashboard.

### Gap 2: Peak Window Parameterization Not Consumed

**Severity:** MODERATE  
**Affected Requirements:** SCHED-03 (user can adjust peak window from 3–6 hours)  
**Impact:** UI allows users to adjust `peak_window_hours` in Overrides panel, but the value is completely ignored. Peak detection always uses hardcoded 4-hour window.

**Root Cause:**  
- ✓ `peak_window_hours` is stored in `app_meta` table
- ✓ Dashboard Overrides panel shows input field to adjust it (Phase 5 verification)
- ✗ `peakDetector()` function is not parameterized; ignores `app_meta` value and always uses `PEAK_WINDOW_HOURS = 4`

**Current State:**
- ✓ Phase 5 (Dashboard) confirms Overrides UI panel exists and saves to app_meta
- ✓ Phase 2 (Algorithm) verifies peakDetector logic
- ✗ No connection between saved `peak_window_hours` and peakDetector parameter

**Fix Required (one of two options):**

**Option A (Recommended):** Parameterize peakDetector
```typescript
async function peakDetector(config: Config, windowHours?: number) {
  const hours = windowHours ?? 4 // default
  // ... use hours instead of hardcoded 4
}
```
Then in scheduler: `peakDetector(config, meta.get('peak_window_hours'))`

**Option B:** Remove UI field  
Delete `peak_window_hours` input from Overrides panel if feature is not yet implemented.

**Verification:** After fix, user adjusts peak window to 5 hours → scheduler uses 5-hour window on next run → detection output changes.
