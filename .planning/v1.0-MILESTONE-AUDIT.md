## Critical Gaps — CLOSED ✓

### Gap 1: Setup Gate Middleware Not Mounted

**Status:** ✓ CLOSED in Phase 9

**Original Issue (v1.0 audit):**  
`src/proxy.ts` contained setup gate logic but was never registered as Next.js middleware. Users bypassed the setup wizard on first boot.

**Phase 9 Fix:**  
Created `src/middleware.ts` exporting setupGate from proxy.ts with proper Next.js middleware pattern.

**Integration Verified:**
- Middleware auto-detects on startup
- Routes requests through setupGate function
- Reads `setup_complete` flag from SQLite app_meta
- Redirects to `/setup` if unset
- Allows dashboard access after setup completes
- **Affected Requirements:** INSTALL-01, INSTALL-02, INSTALL-03

### Gap 2: Peak Window Hours Not Consumed

**Status:** ✓ CLOSED in Phase 9

**Original Issue (v1.0 audit):**  
`peak_window_hours` stored in app_meta but ignored by peakDetector, which was hardcoded to 4 hours. Dashboard override had no effect.

**Phase 9 Fix:**  
Added `windowHours: number = 4` parameter to peakDetector signature. Scheduler reads peak_window_hours from app_meta and passes to peakDetector on every recompute.

**Integration Verified:**
- Dashboard ScheduleOverridesPanel can set 3–6 hour window
- PATCH /api/app-meta writes to app_meta.peak_window_hours
- recomputeSchedule reads value and passes to peakDetector
- Peak detection slides correct window size
- Schedule fires update accordingly
- Test coverage includes windowHours=5 scenario
- **Affected Requirements:** SCHED-03

## E2E Flow Verification

**All 6 critical user flows verified complete and wired:**

### Flow 1: First-Run Setup ✓
- User visits / → middleware redirects to /setup (INSTALL-02)
- Setup wizard collects OAuth token, usage auth, timezone, GCS bucket
- POST /api/setup writes env, sets setup_complete='true' (INSTALL-02)
- Dashboard accessible after (INSTALL-01, INSTALL-03)

### Flow 2: Data Collection → Peak Detection ✓
- UsageCollector polls endpoint with user auth (cookie or bearer)
- Response normalized and stored to usage_snapshots (DATA-01, DATA-02)
- GET /api/dashboard queries snapshots, calls peakDetector with windowHours from app_meta (SCHED-02, SCHED-03)
- OptimalScheduleCard renders peak block and 5 fire times (UI-01)

### Flow 3: Dashboard Override → Recompute ✓
- User edits peak_window_hours (3–6 range) in ScheduleOverridesPanel
- PATCH /api/app-meta writes to app_meta
- Handler calls recomputeSchedule (UI-02, UI-03)
- Scheduler reads windowHours, passes to peakDetector
- Schedule fires update and display changes (SCHED-03)

### Flow 4: Scheduled Sending (60-Second Tick) ✓
- Scheduler registered in instrumentation.ts
- setInterval(runTick, 60000) fires on schedule
- Detects due fires, calls send() for each
- send() spawns `claude -p` with 60s timeout (SEND-01, SEND-02, SEND-04)
- Every attempt logged to send_log (DATA-03)
- Dashboard displays in SendHistoryPanel (UI-04)

### Flow 5: Manual Send ✓
- SendNowButton triggers POST /api/send-now
- Route calls send() with scheduled_for=NULL (SEND-05)
- Entry written to send_log and displayed in dashboard
- **Requirements:** SEND-05, UI-05

### Flow 6: Pause/Resume ✓
- PauseToggle calls PATCH /api/app-meta with paused='true'
- Scheduler reads paused flag and skips sends (SCHED-12)
- User resumes: paused='false' → sends resume on next tick
- **Requirements:** SCHED-12, UI-06
