```
│                                         │

│                                         │
├─────────────────────────────────────────┤
│  SECTION 2: Settings                    │
│  ─────────────────────────────────────  │

│  Field: GCS Bucket [text input]         │
│                                         │
├─────────────────────────────────────────┤

```

### Form Interactions

1. **Field validation (client-side):**
   - OAuth token: required, any non-empty string accepted (validation deferred to server test)
   - Session cookie / Bearer token: required, dependent on radio selection
   - Timezone: optional, default `America/Los_Angeles`
   - GCS bucket: optional, empty string is valid

2. **Radio toggle (Cookie vs Bearer):**
   - Clicking "Session Cookie" shows cookie input, hides bearer input
   - Clicking "Bearer Token" shows bearer input, hides cookie input
   - Only one is collected on submit

3. **Submit flow:**
   - Button text changes to "Setting up..." on click
   - Button is disabled during submission
   - On success: show "✓ Setup complete — redirecting to dashboard" (3s delay, then redirect to `/`)
   - On error: display error message in red text below CTA, button re-enables for retry

### Component Patterns

**Matching existing dashboard:**

- **Section containers:** `rounded-xl p-6` background `--bg-surface`, border `1px solid --border-subtle`
- **Form groups:** `space-y-4` gap between field labels and inputs
- **Text inputs:** `px-4 py-2.5` padding, `rounded-lg` corners, `border-2 --border-subtle` default → `border-2 --accent` on focus, `font-mono` for token/cookie fields
- **Labels:** `text-sm` weight 600, color `--text-primary`
- **Help text:** `text-xs`, color `--text-tertiary`, margin-top `8px`
- **Error text:** `text-xs`, color `--danger`, margin-top `8px`
- **CTA button:** `px-6 py-3`, `rounded-lg`, background `--accent`, color `--bg-base`, weight 600, transitions on hover/active
- **Radio button group:** Use native HTML radio inputs, styled with label wrapping

## Accessibility & Usability

### Form Accessibility

- All inputs have associated `<label>` elements with `htmlFor` binding
- Required fields marked with `*` after label text
- Error messages announced via role="alert" after submission
- Focus indicators visible (focus border uses `--accent`)
- Tab order follows source order (top to bottom)
- Radio button labels clickable (not just the button itself)

### Usability Constraints

- **Non-technical user:** No jargon. "Session cookie" explained as "the method your account uses."
- **Copy clarity:** Every input has inline help text explaining where to get the value
- **Single page:** No multi-step wizard, no progress bar. All fields visible at once reduces cognitive load.
- **30-minute SLA:** Form must load in <2s, submit in <5s even on slow network (progress indicator during save)

## Router & Middleware

### Route Behavior

- **GET /setup:** Accessible only if `app_meta.setup_complete` is NOT `'true'`. If already setup, redirect to `/` (dashboard).
- **POST /api/setup:** Accepts JSON body with `oauth_token`, `auth_method` ('cookie' | 'bearer'), `auth_value`, `timezone`, `gcs_bucket`. Returns `{ success: true }` or `{ success: false, error: "message" }`.
- **Middleware on root route (/):** Checks `app_meta.setup_complete`. If missing or `'false'`, redirect to `/setup`. Otherwise, render dashboard.

### Startup Flow

1. Installer writes `app_meta.setup_complete='false'` to DB
2. Installer starts systemd service
3. User visits app in browser
4. Root middleware detects `setup_complete='false'`, redirects to `/setup`
5. `/setup` form displays
6. User completes form, submits
7. POST /api/setup validates and writes to `/etc/claude-sender.env` (via sudo helper)
8. POST /api/setup writes `app_meta.setup_complete='true'`
9. POST /api/setup response includes redirect hint: `{ success: true, redirect: "/" }`
10. Client redirects to `/`
11. Root middleware sees `setup_complete='true'`, renders dashboard
