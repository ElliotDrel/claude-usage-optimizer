<objective>
Create the privileged helper script (`scripts/write-env.sh`) that safely receives staged configuration from the Next.js API and writes it to the system env file with proper permissions, then restart the service. This script is invoked via `sudo` from the API route but accepts **no arguments** and reads from a fixed file path only — preventing any injection attacks.

Also document the sudoers configuration that the installer will create to allow the `claude-tracker` service user to invoke this helper.

**Purpose:** Separate privilege escalation (root action of writing `/etc/claude-sender.env`) from the API code (unprivileged Next.js process). The no-argument design ensures the API cannot pass malicious input to the helper.

**Output:** Executable `scripts/write-env.sh` committed to the repo; documented sudoers entry created by installer.
</objective>

  <action>
Create `scripts/write-env.sh` (755-mode executable) as a minimal privileged helper that reads from a staging file and writes to the system env file. The script MUST:

1. **Strict mode and argument validation:**
   - `#!/bin/bash` shebang
   - `set -e` to exit on errors
   - At the top: check `[ $# -ne 0 ]` and exit 1 if any arguments are passed (prevents injection)
   - Print error: `echo "Error: write-env.sh accepts no arguments" >&2`

2. **Fixed paths (no user input, all hardcoded):**
   - `STAGING_FILE="/opt/claude-usage-optimizer/data/.env-staging"`
   - `TARGET_FILE="/etc/claude-sender.env"`
   - Both paths absolute; no environment variables or argument-based paths

3. **Validate staging file exists:**
   - Check: `[ ! -f "$STAGING_FILE" ]`
   - If missing: print error `"staging file not found at $STAGING_FILE"` to stderr, exit 1

4. **Copy and set permissions:**
   - Copy staging file to target: `cp "$STAGING_FILE" "$TARGET_FILE"`
   - Set restrictive mode: `chmod 600 "$TARGET_FILE"` (only root can read)
   - Set ownership: `chown root:root "$TARGET_FILE"` (just to be explicit)

5. **Restart service:**
   - `systemctl restart claude-tracker`
   - Wrap in if/else to handle potential momentary failures gracefully
   - If restart fails: print warning but don't exit (env was written, restart might be transient)
   - Exit code 0 on success (whether or not restart succeeded, as long as file was written)

6. **Clean up staging file:**
   - `rm -f "$STAGING_FILE"` (force remove, don't error if not present)

7. **Exit and messaging:**
   - Exit 0 on success
   - Print to stderr on important operations (standard for admin scripts)
   - Print success message: `echo "Setup complete: env written and service restarted" >&2`

**Per D-07 security design:**
- NO arguments accepted (check `[ $# -ne 0 ]` at top)
- NO user input interpolated into commands
- Fixed paths only (staging file path and target path are hardcoded)
- Simple linear flow: validate → copy → chmod → restart → cleanup
- All error conditions print to stderr (`>&2`) not stdout

**Example structure:**
```bash
#!/bin/bash
set -e

if [ $# -ne 0 ]; then
  echo "Error: write-env.sh accepts no arguments" >&2
  exit 1
fi

STAGING_FILE="/opt/claude-usage-optimizer/data/.env-staging"
TARGET_FILE="/etc/claude-sender.env"

if [ ! -f "$STAGING_FILE" ]; then
  echo "Error: staging file not found at $STAGING_FILE" >&2
  exit 1
fi

cp "$STAGING_FILE" "$TARGET_FILE"
chmod 600 "$TARGET_FILE"
chown root:root "$TARGET_FILE"

if systemctl restart claude-tracker; then
  echo "Service restarted successfully" >&2
else
  echo "Warning: systemctl restart returned non-zero, but env file was written" >&2
fi

rm -f "$STAGING_FILE"
exit 0
```

**Important notes:**
- All variable expansions are in double quotes: `"$STAGING_FILE"`, `"$TARGET_FILE"`
- No command substitution or parameter expansion that could be exploited
- Error messages go to stderr (>&2) for proper logging
- Script is deliberately simple and linear (easy to audit)

This task is informational — documenting the sudoers entry created by the installer in scripts/install.sh. The installer already creates this file; this task ensures it's correct and documented.

**Sudoers entry location:** `/etc/sudoers.d/claude-tracker` (created by installer at step 12)

**Required entry content:**
```
claude-tracker ALL=(ALL) NOPASSWD: /opt/claude-usage-optimizer/scripts/write-env.sh
```

**Breakdown:**
- `claude-tracker` — username (service user created by installer)
- `ALL=(ALL)` — allowed on all hosts, as all users (in this case effectively root via sudo)
- `NOPASSWD:` — allow execution without password (safe because entry is narrowly scoped)
- `/opt/claude-usage-optimizer/scripts/write-env.sh` — exactly one script, no wildcards, no `ALL` commands

**File permissions:** Mode 440 (read-only for owner and group, no write, no world-readable)
- Set by installer: `chmod 440 /etc/sudoers.d/claude-tracker`

**Why narrow scope is secure:**
- The helper script (`write-env.sh`) itself has no-argument validation
- Sudoers entry allows only this ONE script, not a directory or pattern
- No shell operators (no `|`, `>`, `&&`, etc.) in the command path
- Even if the API passes a malicious argument to `sudo`, the sudoers entry won't execute it (command is exact)

**Verification by installer:**
- Installer writes the entry: `cat > /etc/sudoers.d/claude-tracker << 'EOF' ... EOF`
- Installer sets mode: `chmod 440 /etc/sudoers.d/claude-tracker`
- No separate validation step needed; `chmod` ensures permissions are correct

**Per RESEARCH.md anti-patterns:**
- NOT `NOPASSWD: ALL` (would allow any command)
- NOT `NOPASSWD: /bin/bash` (would allow arbitrary shell)
- NOT `NOPASSWD: /opt/.../*` (wildcards not needed, specific script only)
- Correctly scoped: `NOPASSWD: /opt/claude-usage-optimizer/scripts/write-env.sh` (exactly this script)

This documentation ensures:
1. The installer creates the correct entry (verified in step 12 of scripts/install.sh)
2. The API can safely invoke the helper via sudo (fixed path, narrow scope)
3. No injection risk (helper validates no args, sudoers entry is specific)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API process → sudo helper | Invokes helper with NO arguments; all secrets in file, not args |
| Helper ← staging file | Reads from fixed path only; validates file exists before use |
| Helper → /etc/claude-sender.env | Writes with restrictive mode (600); root:root ownership |
| Sudoers entry → helper | Narrow scope: only this script, no wildcards, no ALL commands |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege / Shell Injection | Sudo invocation with args | mitigate | No-argument design enforced in script; helper rejects any args with `[ $# -ne 0 ]` |
| T-07-02 | Tampering | Staging file source | mitigate | Helper validates file exists; fixed path only (no user input to path) |
| T-07-03 | Information Disclosure | Env file world-readable | mitigate | Mode 600 (root only); ownership root:root; staging file deleted after merge |
| T-07-04 | Elevation of Privilege | Sudoers entry too broad | mitigate | Entry is scoped to exactly one script; no wildcards, no `ALL` commands |
| T-07-05 | Elevation of Privilege | API passes malicious arg to sudo | mitigate | Helper rejects all arguments; sudoers entry won't run with unexpected args |
| T-07-06 | Denial of Service | Service restart fails | mitigate | Env file is written before restart; restart failure doesn't prevent setup |
</threat_model>
