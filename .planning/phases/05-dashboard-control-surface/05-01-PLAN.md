<objective>
Extend the dashboard API layer to surface schedule state and send history. Add a `PATCH /api/app-meta` endpoint that atomically writes override values, immediately recomputes the schedule, and returns the new schedule to the client. This is the foundation for the UI control surface built in subsequent plans.

Purpose: Enables the UI to display schedule state (peak block, fire times, send history) and to mutate that state (override writes → recompute).

Output: 
- Updated `DashboardData` type with schedule + send history fields
- Extended `GET /api/dashboard` returning schedule data alongside existing usage data
- New `PATCH /api/app-meta` endpoint for override writes with inline schedule recompute
</objective>

    2. Implement PATCH handler:
       ```typescript
       export async function PATCH(req: NextRequest) {
         try {
           const body = await req.json() as { key: string; value: string };
           if (!body.key || body.value === undefined) {
             return NextResponse.json(
               { error: "Missing key or value" },
               { status: 400 }
             );
           }
         
           const config = getConfig();
           
           // 1. Write the override to app_meta
           setAppMeta(config, body.key, body.value);
           
           // 2. Immediately recompute the schedule based on new state
           await recomputeSchedule(config);
           
           // 3. Return the new schedule_fires to the client
           const meta = getAppMeta(config);
           const newScheduleFires = meta.get("schedule_fires");
           
           return NextResponse.json({
             success: true,
             key: body.key,
             value: body.value,
             scheduleFires: newScheduleFires ? JSON.parse(newScheduleFires) : null,
           });
         } catch (err) {
           console.error("[app-meta]", err);
           return NextResponse.json(
             { error: err instanceof Error ? err.message : "Unknown error" },
             { status: 500 }
           );
         }
       }
       ```

       - `paused` ("true" or "false" string)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → PATCH /api/app-meta | Untrusted override values (user_timezone string, peak_window_hours number, etc.) |
| Database writes (app_meta) | No schema validation at DB layer; app_meta is untyped key-value |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | PATCH /api/app-meta request body | Mitigate | Validate key names against whitelist (schedule_override_start_time, peak_window_hours, anchor_offset_minutes, default_seed_time, user_timezone, paused). Reject unknown keys with 400 response. |
| T-05-02 | Tampering | app_meta values (invalid formats) | Mitigate | Let recomputeSchedule validation catch format errors (e.g., invalid IANA timezone string); return 500 with error message. User sees error and can retry. |
| T-05-03 | Information Disclosure | GET /api/dashboard response | Accept | Response contains schedule state and send history. No PII beyond what's in send_log.response_excerpt (user's truncated question+response). Response is same auth boundary as existing dashboard (local-only in prod). |
| T-05-04 | Denial of Service | PATCH /api/app-meta with rapid requests | Accept | In-process scheduler handles recompute; no external API calls. Recompute is O(snapshots), bounded by historical data size. No rate limiting needed for Phase 5; monitor in Phase 6. |

</threat_model>
