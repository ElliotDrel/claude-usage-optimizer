  timestamp: string
  status: string
  endpoint: string
  responseStatus: number
  rawJson: string | null
  errorMessage: string | null

New SCHEMA string — 7-column table + 2 indexes + app_meta table:
  usage_snapshots: id, timestamp, status, endpoint, response_status, raw_json, error_message
  idx_snapshots_timestamp ON usage_snapshots(timestamp)
  idx_snapshots_status ON usage_snapshots(status)
  app_meta: key TEXT PRIMARY KEY, value TEXT NOT NULL

migrateToSimplifiedSchema logic:
  1. Check app_meta for key="schema_version", value="simplified-v1" -> if found, return immediately (idempotent)
  2. Inside a db.transaction(), run PRAGMA table_info(usage_snapshots) to check if auth_mode column exists
  3a. If auth_mode exists (old schema): CREATE usage_snapshots_new (7 cols) -> INSERT SELECT 7 surviving cols -> DROP usage_snapshots -> RENAME -> CREATE INDEX x2
  3b. If auth_mode absent (fresh install): skip the table rebuild (SCHEMA already created correct table)
  4. In both cases: upsert app_meta key="schema_version" value="simplified-v1" and key="migrated_at" value=new Date().toISOString()

getDb changes: remove the MIGRATIONS for-loop, remove migrateExtraUsageMoneyToDollars call, add migrateToSimplifiedSchema(db) after db.exec(SCHEMA)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite migration | The migrator reads/writes a local SQLite file; no network or user input involved |
| raw_json storage | raw_json is stored as-is from the API response string; safeParseRaw on the read side wraps JSON.parse in try/catch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | migrateToSimplifiedSchema DROP TABLE | mitigate | Entire migration runs inside a single db.transaction(); if any step throws, SQLite rolls back atomically — no partial migration |
| T-02-02 | Information Disclosure | raw_json verbatim storage | accept | raw_json is the Claude API response stored locally on the user's own machine; no additional PII exposure vs old schema |
| T-02-03 | Denial of Service | migration on large table | accept | Local desktop app with bounded row count; O(n) copy step is acceptable |
| T-02-04 | Elevation of Privilege | SQL via insertSnapshot data param | accept | All values are bound via parameterized queries — no string interpolation into SQL |
</threat_model>
