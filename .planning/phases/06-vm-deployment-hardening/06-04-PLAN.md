**Structure (8 sections in this order):**

1. **Title and Overview** (brief, ~3 sentences)
   - What: Single systemd service running Claude Usage Optimizer on a GCP e2-micro VM
   - Why: Durability, production-ready, accessible via SSH tunnel
   - Time estimate: ~30 minutes end-to-end for a non-technical user

2. **Prerequisites** (what user must have before starting)
   - GCP account with billing enabled
   - Claude Code OAuth token (from `claude setup-token` output)
   - Terminal/SSH client (for connecting to VM)
   - Basic comfort with copy-paste (no deep CLI knowledge required)

3. **Provision VM** (step-by-step GCP console instructions)
   - Create e2-micro instance in us-central1, Ubuntu 22.04 LTS, 30 GB pd-standard
   - Add 2 GB swap file (include copy-pasteable commands: `sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile && ...`)
   - Create GCS bucket for backups (include gsutil command: `gsutil mb -c STANDARD -l us-central1 gs://YOUR_BUCKET_NAME`)
   - Get SSH key and save it
   - Test SSH connection

4. **Deploy App** (clone repo, install deps)
   - SSH into VM
   - Clone repo: `git clone https://github.com/...`
   - Install Node.js 20: `curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash - && sudo apt-get install -y nodejs`
   - Install gcloud CLI: `curl https://sdk.cloud.google.com | bash && exec -l $SHELL`
   - cd into app directory
   - npm install
   - npm run build

5. **Configure Secrets** (OAuth token, env file, permissions)
   - Create /etc/claude-sender.env from template (paste the example from /etc/claude-sender.env.example, fill in OAuth token and GCS bucket)
   - Copy-pasteable block: `sudo tee /etc/claude-sender.env > /dev/null << 'EOF'` followed by the template content
   - Set file permissions: `sudo chmod 600 /etc/claude-sender.env` with verification: `ls -l /etc/claude-sender.env` should show `-rw-------`

6. **Install and Start Service** (systemd setup)
   - Create non-root user: `sudo useradd -r -s /bin/bash -d /opt/claude-usage-optimizer claude-tracker`
   - Copy systemd unit file (from /etc/systemd/system/claude-tracker.service in repo)
   - Copy-pasteable: `sudo cp /opt/claude-usage-optimizer/claude-tracker.service /etc/systemd/system/`
   - Reload systemd: `sudo systemctl daemon-reload`
   - Enable service: `sudo systemctl enable claude-tracker`
   - Start service: `sudo systemctl start claude-tracker`
   - Verify running: `sudo systemctl status claude-tracker`

7. **Verify Deployment** (post-deploy checklist, §QUAL-03)
   Include a structured verification section with numbered steps. Each step must be copy-pasteable and have an expected output:
   
   - **Service Health:** `sudo systemctl status claude-tracker`
     Expected: "active (running)" in green
   
   - **CLI Authentication:** `sudo journalctl -u claude-tracker -n 20`
     Expected: "[scheduler] Scheduler started" and "[instrumentation] Collector started" in recent logs
   
   - **Scheduler Ticking:** `sudo sqlite3 /opt/claude-usage-optimizer/data/usage.db "SELECT value FROM app_meta WHERE key='last_tick_at';" | head -1`
     Expected: Recent ISO timestamp (within last 5 minutes)
   
   - **First Send Logged:** `sudo sqlite3 /opt/claude-usage-optimizer/data/usage.db "SELECT status FROM send_log ORDER BY fired_at DESC LIMIT 1;"`
     Expected: 'ok' or 'error' or 'timeout' (any status value)
   
   - **Nightly Backup:** `gcloud storage ls gs://YOUR_BUCKET_NAME/backups/daily/ 2>/dev/null | head -1`
     Expected: At least one .db.gz file (will appear after 04:15 UTC)

8. **Configure Backups** (GCS lifecycle rule)
   - Explain: Backups run nightly at 04:15 UTC and upload to GCS
   - Set 30-day lifecycle rule: `gsutil lifecycle set - gs://YOUR_BUCKET_NAME << 'EOF'` followed by JSON lifecycle config (delete objects >30 days old)
   - Verify: `gsutil lifecycle get gs://YOUR_BUCKET_NAME`

9. **Configure Notifications** (optional Discord webhook)
   - Create Discord webhook URL (link to Discord docs)
   - Copy webhook URL
   - Via dashboard: navigate to `http://localhost:3018` (via SSH tunnel), go to Overrides panel, paste webhook URL into notification field (if UI has this field; if not, use SQL: `sqlite3 ... INSERT INTO app_meta ...`)
   - Test: trigger manual send, verify Discord receives notification
   - Can also be configured later (opt-in)
