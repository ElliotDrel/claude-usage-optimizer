Create a new file src/lib/backup.ts implementing the in-process backup job following the pattern from RESEARCH.md and PATTERNS.md.

The file must:

1. **Import structure** (following project conventions):
   - Node.js builtins: `promises as fs`, `createReadStream`, `createWriteStream` from `node:fs`, `zlib`
   - Third-party: `Storage` from `@google-cloud/storage`, `type Database` from `better-sqlite3`
   - Local: `backupDatabase` from `./db`, `getConfig` from `./config`

2. **backupToGcs(db: Database.Database): Promise<void>** — performs one backup cycle:
   - Generate timestamp (ISO format with colons/dots removed for filename safety): `usage-YYYYMMDDTHHMMSSZ.db`
   - Temp paths: `/tmp/usage-{timestamp}.db` and `/tmp/usage-{timestamp}.db.gz`
   - Call `backupDatabase(config, backupPath)` to create local backup
   - Pipe the backup through gzip to create .db.gz file using Node streams
   - Read `GCS_BACKUP_BUCKET` env var; throw error if not set
   - Create Storage() client and upload gzipped file to `gs://bucket/backups/daily/{timestamp}.db.gz`
   - Delete temp files after upload
   - **Error handling:** Wrap entire function in try/catch. On error, log to console.error() but do NOT rethrow. Per D-02, backup failures are non-fatal.
   - Log progress: "[backup] starting..." → "[backup] compressing..." → "[backup] uploading..." → "[backup] uploaded to gs://..."

3. **scheduleDaily(utcTime: string, job: () => Promise<void>): NodeJS.Timeout** — utility helper:
   - Parse `utcTime` as "HH:MM" format (e.g., "04:15")
   - setInterval every 60 seconds, check if current UTC time matches target hour and minute
   - If match, invoke job asynchronously with `.catch()` to prevent interval from throwing
   - Return the interval ID for cleanup

4. **startBackupJob(db: Database.Database): { stop: () => void }** — main export:
   - Call `scheduleDaily("04:15", async () => { await backupToGcs(db); })`
   - Return object with `stop` function that calls `clearInterval()`
   - This mirrors the scheduler registration pattern exactly (see src/lib/scheduler.ts lines 438-467)

## Trust Boundaries

| Boundary | Description | Applies to Phase 6? |
|----------|-------------|---------------------|
| GCS API (outbound HTTPS) | Backup upload to Google Cloud Storage | Yes — T-06-02 |
| Local filesystem (/tmp) | Temporary backup files before gzip | Yes — T-06-03 |
| GCS credentials (env var or ADC) | Authentication for GCS bucket access | Yes — T-06-03 |
| SQLite database on disk | Source of backup data | Yes (existing, Phase 1) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | OAuth token in /etc/claude-sender.env | mitigate | File permissions set to mode 600 (readable by claude-tracker user only); documented in HOSTING-STRATEGY.md. Phase 6 enforces via systemd unit file docs. |
| T-06-02 | Information Disclosure | GCS backup upload via HTTP | mitigate | `@google-cloud/storage` SDK uses HTTPS only; Google Cloud Storage is encrypted at rest (managed keys). Backup objects inherit bucket encryption policy. |
| T-06-03 | Information Disclosure | Temp backup files in /tmp | mitigate | Temp files are deleted immediately after gzip+upload (src/lib/backup.ts backupToGcs line 395-396, `await fs.unlink()`). Window of exposure is <10 seconds per backup cycle (04:15 UTC daily). |
| T-06-04 | Information Disclosure | GCS credentials not found at runtime | mitigate | If `GOOGLE_APPLICATION_CREDENTIALS` env var is not set and ADC is not available, backup logs `[backup] failed: GCS_BACKUP_BUCKET not configured` and continues (non-fatal per D-02). Phase 7 installer will provision credentials; Phase 6 documents manual setup. |
| T-06-05 | Denial of Service | Backup job contends with send fires at 04:15 UTC | accept | Backup runs at 04:15 UTC daily; user's sends are distributed across 5 fires (spaced 5 hours apart). Probability of collision at exact minute is low. If observed, backup time can be adjusted (documented in RESEARCH.md Pitfall 2). SQLite WAL mode mitigates write contention. |
| T-06-06 | Information Disclosure | Webhook URL stored in app_meta (plaintext) | accept | Per decision D-07, webhook URL lives unencrypted in SQLite. Acceptable trade-off for simplicity (no encryption layer). Risk: if VM is compromised, webhook URL can be exfiltrated. Mitigation: regenerate webhook URL periodically; bind webhook to single Discord channel. Documented in HOSTING-STRATEGY.md. |

</threat_model>
