Create a new file /etc/systemd/system/claude-tracker.service with the systemd unit file content.

The unit file must:

1. **[Unit] section:**
   - Description: "Claude Usage Optimizer Tracker"
   - After: "network.target" (ensure network is ready before starting)

2. **[Service] section:**
   - Type: "simple" (process-based, not forking)
   - User: "claude-tracker" (non-root user, D-09)
   - Group: "claude-tracker" (same group)
   - WorkingDirectory: "/opt/claude-usage-optimizer" (repo root, where package.json lives)
   - EnvironmentFile: "/etc/claude-sender.env" (D-09, loads HOSTNAME, PORT, CLAUDE_CODE_OAUTH_TOKEN, NODE_ENV, GCS_BACKUP_BUCKET)
   - ExecStart: "/usr/bin/npm run start:prod" (production server startup command, D-09)
   - Restart: "always" (D-09, recover from crashes)
   - RestartSec: "5" (D-09, wait 5 seconds before restarting)
   - StandardOutput: "journal" (log to systemd journal)
   - StandardError: "journal" (stderr to systemd journal)

3. **[Install] section:**
   - WantedBy: "multi-user.target" (enable service for multi-user mode)

4. **Security and logging notes:**
   - User=claude-tracker ensures app doesn't run as root (standard practice)
   - StandardOutput/StandardError=journal makes logs accessible via `journalctl -u claude-tracker`
   - EnvironmentFile loads env vars at startup; next systemd requires explicit list if ExecStart needs variables, but npm handles this automatically

**Important:** Do NOT set Environment=HOSTNAME or Environment=PORT in the unit file. These must come from EnvironmentFile (/etc/claude-sender.env) so they can be updated without redeploying the unit file.

**Content reference:** Use exact template from 06-PATTERNS.md lines 512-531. The template is canonical.

<threat_model>
## Trust Boundaries

| Boundary | Description | Applies to Phase 6? |
|----------|-------------|---------------------|
| /etc/claude-sender.env (local filesystem) | Env file with OAuth token and GCS credentials | Yes — T-06-01 |
| systemd service process (privileged) | Service manager runs as root to start the app | Yes — T-06-11 |
| Network binding (127.0.0.1:3018) | Localhost-only listening address | Yes — T-06-12 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering / Disclosure | /etc/claude-sender.env permissions too loose | mitigate | File must have mode 600 (rw-------) so only claude-tracker user can read. Documented in env file template with security warning. HOSTING-STRATEGY.md includes `sudo chmod 600` step. |
| T-06-02 | Tampering / Disclosure | OAuth token exposed via env var or process listing | mitigate | Token stored in env file with restrictive perms (600); at runtime, Next.js reads via process.env (not passed as command-line arg). `ps` command will not show env values (kernel protects). |
| T-06-11 | Elevation | Systemd starts service as non-root, but socket is root-owned initially | mitigate | Service runs as claude-tracker user (non-root). Unit file specifies User=claude-tracker. Systemd binds the listening socket before dropping privs (standard practice). App cannot escalate back to root. |
| T-06-12 | Authorization | Binding to public interface (0.0.0.0) instead of 127.0.0.1 | mitigate | Unit file loads HOSTNAME=127.0.0.1 from env file. Dashboard has no auth layer; binding to 127.0.0.1 enforces network-level access control. Documented in env file: "never 0.0.0.0". |
| T-06-13 | Information Disclosure | systemd journal logs contain sensitive errors | accept | StandardOutput=journal and StandardError=journal mean all stdout/stderr goes to journalctl. If errors contain tokens or credentials, they could be exposed. Mitigated by code that avoids logging secrets; documented in code review. |

</threat_model>
